Legacy MDD/AIMDD/IVDD: globalModelDescription & FLD-UDID-22

Frage von Maik & Antwort von Zeno — Status quo + Umstellungsplan sobald GS1-Rule 097.116 auf MDR/IVDR-only verfeinert ist.

eudamed2firstbase v1.0.44 · 2026-05-01

Frage von Maik

Mir fällt was auf: Wie hatten wir das Problem bei MDD/AIMDD/IVDD gelöst, wenn in EUDAMED das Attribut Model Name (FLD-UDID-22) nicht in globalModelDescription reingemappt bekommst, weil die globalModelNumber dann auch gefüllt sein muss, aber wir können diese wegen firstbase rule 097.116 (Prüfziffervalidierung) nicht mit B-GTIN values mappen?

Ich arbeite mit TC an einer Verfeinerung von 097.116 (nur bei MDR/IVDR), weil uns das auch an einer anderen Stelle auf die Füsse fällt. Dann könntest du zukünftig bei MDD/AIMDD/IVDD records globalModelNumber mit B-GTIN (sog. EUDAMED DI) schicken. Und dann müssen aber auch immer die Model Name (FLD-UDID-22) in globalModelDescription gemappt kommen.

Aktueller Status MDD/AIMDD/IVDD

In src/transform_detail.rs (Zeile 600–624):

// 097.095: Legacy devices must not have globalModelNumber
global_model_info: if is_legacy {
    Vec::new()           // ← komplett leer, weder Number noch Description
} else {
    vec![GlobalModelInformation {
        number:       <Basic UDI-DI Code, fallback Primary DI>,
        descriptions: <FLD-UDID-22 deviceName, fallback Primary DI, lang=en>,
    }]
}

D.h. bei MDD/AIMDD/IVDD landet weder globalModelNumber noch globalModelDescription im Output. FLD-UDID-22 (Model Name) geht aktuell bei Legacy verloren.

Als Ersatz emittieren wir bei Legacy ein MODEL_NUMBER in additionalTradeItemIdentification — gefüllt mit deviceModel (FLD-UDID-20), und falls leer mit dem Basic UDI-DI Code (= EUDAMED B-GTIN). Siehe src/transform_detail.rs:244-266. Aber das ist nur Number, nicht Name.

Plan, sobald 097.116 auf MDR/IVDR-only verfeinert ist

Anpassung an einer einzigen Stelle — der if is_legacy { Vec::new() }-Zweig fällt weg, dann gilt bei Legacy derselbe Pfad wie heute bei MDR/IVDR:

  • globalModelNumber ← Basic UDI-DI Code (EUDAMED DI / B-GTIN)
  • globalModelDescription ← FLD-UDID-22 (deviceName) auf en

Alle vier Konverter-Pfade (transform.rs, transform_api.rs, transform_detail.rs, transform_eudamed_json.rs) greifen auf dieselbe Funktion zurück.

Offene Punkte

  • Termin für 097.116-Verfeinerung von TC abwarten
  • Entscheidung: Feature-Flag in config.toml (z. B. legacy_global_model = true) zur stufenweisen Aktivierung, oder Direktumstellung mit Release?
  • Bei Legacy-Devices ohne deviceModel/basic_udi.code: ist MODEL_NUMBER in additionalTradeItemIdentification dann noch nötig oder redundant zu globalModelNumber?
