Firstbase Push Log — Test / Production Segregation

How eudamed2firstbase v1.0.39 guarantees that a Test push and a Production push can never be confused — neither in the database, the filesystem, the report, nor the operator UI.

Document: 2026-04-17  ·  Tool version: 1.0.39  ·  Source of truth: src/gui.rs (push session recording + HTML report)

Why we added this

The GS1 Firstbase Catalogue Item API has two environments with different base URLs but can use the same Publish-To-GLN (e.g. 7612345000527). Before v1.0.39 the push_log table stored the GLN but not the environment, so a row for “I pushed to Test” was indistinguishable from “I pushed to Production” once the in-memory log was gone. That created a real risk of reading a Test result and acting as if it were Production data (or vice versa).

What changed in v1.0.39

Four independent layers now tag every push with its environment, so a mistake on one layer is caught by the others.

Layer 1 — Database columns

ALTER TABLE push_log     ADD COLUMN firstbase_env TEXT NOT NULL DEFAULT '';
ALTER TABLE push_session ADD COLUMN firstbase_env TEXT NOT NULL DEFAULT '';
ALTER TABLE push_session ADD COLUMN api_base      TEXT NOT NULL DEFAULT '';
CREATE INDEX idx_push_log_env     ON push_log(firstbase_env);
CREATE INDEX idx_push_session_env ON push_session(firstbase_env);
  • firstbase_env = 'Test' or 'Production' (legacy rows from before v1.0.39 remain empty and are surfaced as “UNKNOWN ENVIRONMENT” — never silently recoloured).
  • api_base stores the full URL used, so the exact endpoint can be audited even if environment naming changes later.
  • Indexes keep GROUP BY firstbase_env queries fast across millions of rows.

Layer 2 — Filesystem

HTML reports are physically separated into two subdirectories, so Test and Production logs can never land in the same folder:

~/eudamed2firstbase/log/
  ├── firstbase_prod/
  │   └── 14.35_17.04.2026.log.html
  └── firstbase_test/
      └── 11.20_17.04.2026.log.html

Layer 3 — Full-width banner in the HTML report

Every generated report carries a banner spanning the full page at the top. The colour is non-negotiable — you cannot miss which environment the log refers to, even in a quick glance or a WhatsApp preview.

🔒 PRODUCTION — LIVE DATA https://webapi-firstbase.gs1.ch
🔒 TEST ENVIRONMENT https://test-webapi-firstbase.gs1.ch:5443

The report’s summary block also adds Environment and API base rows next to the existing Version / Timestamp / Publish GLN / Accepted / Rejected fields.

Layer 4 — GUI

The WhatsApp section of the GUI now shows two separately coloured buttons:

ButtonFill colourSource directory
Send latest Prod logredlog/firstbase_prod/
Send latest Test logbluelog/firstbase_test/

Each button only reads from its own subdirectory, so you cannot accidentally ship a Test log when you meant Production.

Inspection queries

-- Environment summary across all sessions
SELECT firstbase_env, api_base,
       COUNT(*)                 AS sessions,
       SUM(total_accepted)      AS accepted,
       SUM(total_rejected)      AS rejected
  FROM push_session
 GROUP BY firstbase_env;

-- Was a specific GTIN ever pushed to Production?
SELECT firstbase_env, pushed_at, status, error_code
  FROM push_log
 WHERE gtin = '07612345780313'
 ORDER BY pushed_at;

-- Last 10 Production pushes
SELECT uuid, gtin, pushed_at, status
  FROM push_log
 WHERE firstbase_env = 'Production'
 ORDER BY pushed_at DESC
 LIMIT 10;

Backwards compatibility

Rows and files from before v1.0.39 remain in place, unmodified. The report renders a neutral grey banner “UNKNOWN ENVIRONMENT (legacy log)” for them, and the SQL queries treat them as firstbase_env = ''. They are not retro-fitted, because we cannot know after the fact which environment they targeted — but they stay visible and auditable.
